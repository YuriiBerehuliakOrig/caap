# Compiling to Native Code and WebAssembly

So far programs have run on the evaluator. CAAP also compiles a **typed subset**
of the language to LLVM IR and to WebAssembly, producing fast, freestanding
binaries. This chapter shows the toolchain and what the subset supports. The
backends are themselves stdlib modules — codegen is CAAP, like everything else
above the kernel.

## The Native Subset

Native code needs exact types, so the native path accepts the *typed* subset
from Chapter 8 plus systems features:

- typed `defn`, sized integers and floats, `struct`s, and strings (through the
  runtime ABI);
- `if`, `while`, `bind`, and `ref`/`deref`/`set_ref`;
- bit operations, numeric casts, and global variables;
- `extern` calls, typed pointers, fixed-size arrays, and function pointers;
- memory-mapped I/O (MMIO), inline assembly, and freestanding object emission.

Untyped kernel features that have no fixed machine representation (arbitrary
maps, closures over dynamic values) are not part of the native subset; they live
on the evaluator.

## The Toolchain

Native builds run on the ordinary `stdlib/bootstrap.caap` together with a tool
from `tools/` — each `s2_*` tool loads its backend on demand:

| Tool | Output |
|---|---|
| `tools/s2_emit.caap` | LLVM IR text |
| `tools/s2_build.caap` | a native executable (needs `clang`) |
| `tools/s2_wasm.caap` | WebAssembly text (WAT) |

## A Native Program

The corpus's `native_hello.caap` is a *native-only* program: `println` here is a
runtime operation the native linker resolves, and `main`'s integer result is the
process exit code.

```scheme
; native_hello.caap
(bind main
  (lambda ()
    (do
      (println "hello from stdlib native")
      (println "two\nlines and a back\\slash")
      42)))

(main)
```

### Emitting LLVM IR

```bash
$ caap stdlib/bootstrap.caap tools/s2_emit.caap /abs/path/native_hello.caap
```

produces (abbreviated) exactly this IR:

```llvm
; generated by stdlib.backend.emit.llvm
declare i32 @caap_runtime_print_string_ln({ ptr, i64 })

@.s1 = private unnamed_addr constant [25 x i8] c"hello from stdlib native"

define i32 @main() {
entry:
  %t1 = call i32 @caap_runtime_print_string_ln({ ptr, i64 } { ptr @.s1, i64 25 })
  ...
  ret i32 42
}
```

Notice `println` lowered to a call to `@caap_runtime_print_string_ln`, and the
string became a program global passed as a `{ ptr, i64 }` — CAAP's string ABI.

### Building and Running an Executable

```bash
$ caap stdlib/bootstrap.caap tools/s2_build.caap /abs/path/native_hello.caap ./hello
$ ./hello
hello from stdlib native
two
lines and a back\slash
$ echo $?
42
```

`s2_build` emits the IR, then drives `clang` to compile and link it against the
runtime library, yielding a self-contained binary whose exit code is `main`'s
return value — `42`.

## The Runtime: `caap-sys-runtime-ffi`

The `caap_runtime_*` symbols the IR calls are provided by the **system runtime**.
The logic lives in the `caap-sys-runtime` crate; the link unit that exposes it
with a C ABI (as a static/`cdylib` library) is the small `caap-sys-runtime-ffi`
crate. There is no separate hand-written C runtime — a compiled CAAP program
links the same runtime the interpreter uses, just reached through `extern "C"`
entry points. Chapter 14 covers what that runtime offers.

## WebAssembly

```bash
$ caap stdlib/bootstrap.caap tools/s2_wasm.caap /abs/path/program.caap > program.wat
$ wat2wasm program.wat -o program.wasm   # then run with node, wasmtime, …
```

The WebAssembly backend (`stdlib.backend.emit.wasm`) is a sibling of the LLVM backend
and emits WAT for the wasm-meaningful parts of the subset: structs, unions,
enums, globals, `extern`, function pointers, linear-memory arrays, and string
`println`. Features with no wasm meaning (MMIO, inline asm) are correctly
rejected.

## Bare-Metal: Cross-Compiling to Cortex-M (Phase 0)

The same subset runs on real microcontrollers. Nothing new enters the kernel:
the bare-metal capability is **target policy** in `stdlib.backend.emit.llvm` /
`stdlib.backend.driver` plus two declaration spellings in `stdlib.frontend.clike` —
all of it above the 3-node kernel (see `docs/design-bare-metal.md` for the full
audit and the principle gate).

Three pieces make it work:

1. **Target triple + datalayout** — `stdlib.backend.emit.llvm` is host-neutral by default
   (it emits *no* `target` lines, so the host build is unchanged). For a cross
   build, `set_target!` pins the triple and datalayout; `stdlib.backend.driver`
   holds the `targets` table (`cortex-m3 -> thumbv7m-none-eabi`).
2. **A freestanding build** — `stdlib.backend.driver.compile_freestanding`
   cross-compiles the IR with `clang -target ... -ffreestanding -nostdlib`,
   assembles an ARM asm
   shim, and links a custom linker script. The host runtime is **not** linked.
   For a whole clike *surface* program with its own dependency modules,
   `stdlib.backend.driver` offers a one-call wrapper,
   `compile_surface_freestanding(dir, entry, modules, target, out, opts)`, which
   folds the entire pipeline — lower each module, declare them, lower the entry,
   look up the target, run `compile_freestanding`, and clean up — into a single
   call. The URun example (`examples/urun/ur_build.caap`) drives a cross build
   this way.
3. **System vocabulary in the NMV surface** — a *bodyless* declaration
   `name (params) ret` (no `= { … }`) is an `extern`; a top-level
   `name type = expr` is a mutable `global`. MMIO/pointer/asm are ordinary
   calls (`volatile_write(addr, u32, v)`), because an undeclared type word
   lowers to a bare type name — exactly what the backend wants.

The native head/type vocabulary that all of this leans on lives in one module,
`stdlib.backend.native_meta` — the single source of truth for both prep's gate
vocabulary and the wasm emitter's reject set, so the two stay in lockstep.

By default the gate is permissive: an undeclared type word lowers to a bare type
name (as above). There is also an opt-in **strict** native profile. Both
`compile_freestanding` and `compile_surface_freestanding` honor `opts.mode:
"strict"`, which makes the gate reject any declared type that resolves to no
native or program type. It is off by default; turn it on when you want declared
types validated rather than passed through.

The Phase 0 example (`examples/bare_uart.caap`, authored in
`(surface stdlib.frontend.clike)`) writes `"OK\n"` to the CMSDK UART of QEMU's
`mps2-an385` board:

```bash
$ caap stdlib/bootstrap.caap tools/s2_bare.caap \
      bare_uart.caap out.elf cortex-m3 bare_uart.s bare_uart.ld
$ qemu-system-arm -M mps2-an385 -cpu cortex-m3 -nographic -kernel out.elf
OK
```

Target selection stays bootstrap/tool policy — the CLI never grows a `--target`
flag (principle #7).

> **Phase annotations.** A full bare-metal RTOS path is *implemented and tested*,
> Phases 0–3: Phase 0 (UART, above); **Phase 1** (a cooperative context switch —
> `bare_ctx.caap`, "MABABAB…"); **Phase 2** (a *preemptive* round-robin scheduler
> — `bare_pre.caap`, two NON-yielding busy loops the SysTick timer slices into
> "MAAA…BBB…AAA…"); **Phase 3** (synchronization — `bare_sync.caap`, two tasks
> race to increment a shared counter through a wide critical section, printing
> "MY" when all 200 increments survive; without the lock the same workload prints
> "MN"). Tests: `stdlib_bare_metal_{uart_phase0,context_switch_phase1,
> preemptive_scheduler_phase2,synchronization_phase3}`. Remarkably, **none** of
> them needed a kernel or backend change beyond Phase 0's target policy — not
> the preemptive scheduler (its exception machinery lives in the vector-table asm
> shim reached by hardware) and not synchronization (single-core mutual exclusion
> is interrupt masking via the existing `asm` primitive). Task creation,
> scheduling, and locking are all CAAP over the Phase-0 primitives. The full
> audit + design lives in `docs/design-bare-metal.md`.
